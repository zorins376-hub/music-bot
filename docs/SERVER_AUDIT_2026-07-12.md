# Аудит сервера и кодовой базы — 2026-07-12

## Резюме

Провёл аудит проекта music-bot с фокусом на безопасность сервера и проверку
логов. **Прямой доступ к серверу (`89.169.52.174`) из текущего окружения
невозможен** — исходящий SSH (порт 22) и HTTP к произвольному хосту блокируются
политикой egress-прокси, поэтому live-логи с сервера прочитать не удалось (см.
раздел «Доступ к серверу»). Основной результат аудита — по коду.

**Главная проблема: публичный репозиторий содержал рабочие production-креды**
(root-пароль VPS, токены ботов, пароли и service_role-ключи Supabase). Все
жёстко зашитые секреты вынесены в переменные окружения этим PR, но так как они
уже в истории git публичного репо — **их обязательно нужно ротировать**.

Секреты уже в истории git публичного репо, поэтому **ротация обязательна**:
1. Сменить root-пароль VPS (лучше — перейти на SSH-ключи, отключить
   парольный вход: `PasswordAuthentication no`).
2. Перевыпустить токены обоих ботов через @BotFather (`/revoke`).
3. Сменить пароли обеих баз Supabase.
4. Перевыпустить service_role-ключи Supabase (Dashboard → API → Roll key)
   и положить их в Vault.
5. Рассмотреть очистку истории git (`git filter-repo`) или пересоздание репо,
   так как старые коммиты всё ещё содержат секреты.

## Высокий риск — обход авторизации админки при пустой конфигурации

`webapp/admin_api.py`: если `ADMIN_USERNAME` / `ADMIN_PASSWORD` /
`ADMIN_SESSION_SECRET` не заданы, код лишь писал warning «admin panel
disabled», но фактически **эндпоинты оставались открытыми**: пустой логин
проходил через `hmac.compare_digest("", "")` → выдавался валидный токен на
все админ-действия (бан юзеров, выдача премиума, удаление треков, чтение логов).

**Исправлено в этом PR:** введён флаг `_ADMIN_ENABLED`; при отсутствии любой
из трёх переменных `/login` возвращает 503, а проверка сессии — отказ.

## Средний риск — сетевое экспонирование

- `docker-compose.yml`: сервис `bot` публикует порт `8080` напрямую на хост
  (`ports: "8080:8080"`), nginx закомментирован. То есть TMA webapp **и
  админ-API** доступны по HTTP без TLS и без reverse-proxy. Рекомендация:
  вернуть nginx с TLS (`nginx.conf` уже готов, заменить `YOUR_DOMAIN`), а порт
  8080 не публиковать наружу (`expose` вместо `ports` или биндить на
  `127.0.0.1:8080`).
- Порт `9090` (Prometheus metrics) также публикуется — закрыть от внешнего
  доступа.
- `POSTGRES_PASSWORD: ${POSTGRES_PASSWORD:-changeme}` — дефолт `changeme`.
  Postgres не публикуется на хост (только внутренняя сеть), но пароль всё равно
  задать сильный в `.env`.

## Проверка логов на сервере (выполнить при наличии SSH)

Когда будет прямой доступ к серверу, проверить:

```bash
cd /opt/music-bot
docker compose ps                          # статус контейнеров
docker compose logs --tail=200 bot         # логи бота
tail -n 200 logs/errors.log                # ошибки приложения
grep -iE 'error|traceback|exception' logs/errors.log | tail -50
journalctl -u docker --since '1 day ago' --no-pager | tail -100
last -20 ; grep -i 'failed password' /var/log/auth.log | tail -30   # следы брутфорса SSH
df -h ; free -m                            # диск/память
```

Особое внимание к `auth.log`: так как root-пароль был публичным, проверить
журнал на успешные и неуспешные входы с чужих IP до момента ротации пароля.
